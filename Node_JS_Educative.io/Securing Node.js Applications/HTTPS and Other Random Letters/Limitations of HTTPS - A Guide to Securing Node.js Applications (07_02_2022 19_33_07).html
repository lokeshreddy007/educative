Limitations of HTTPS

Learn about some of the limitations of HTTPS.

We'll cover the following
    • Limitations
    • Certificate types

Limitations#

There are a few limitations when using HTTPS that may make it infeasible in certain circumstances.

Virtual hosts

Under normal configurations, virtual hosts cannot be used with SSL. This is a problem when using shared hosting or simply running multiple sites on the same server. The server can’t determine the host header until the connection has been completed, which requires SSL authentication. Because certificates can have only one host, it will simply not work.

There is an easy way around this! Set up multiple IP addresses and use IP-based hosts instead of the name-based host resolution. I recommend setting up a separate server for secure sites. Generally, if you need HTTPS, you are probably at the point of needing a dedicated server as well.

There are some hosting providers with shared certificates that can be used across hosted sites. This enables you to quickly and cheaply support HTTPS. But the domain needs to reflect the hosting provider’s domain name. For example, instead of:

https://yourApp.com/login

the URL would be something like

https://yourHost.com/yourApp/login

Depending on your application and branding needs, this may or may not be a concern.

Speed

HTTPS connections require SSL handshakes to establish the connection. This makes the overall transfer slower. Once that initial handshake is performed, additional connections only require the encryption and decryption of the content. Subsequent connections aren’t much slower once the initial connection is complete. The performance impact is incredibly low. This is not a valid reason to stop using HTTPS.

Caching

The cache is the secret sauce behind super quick load times. Modern browsers will cache HTTPS content the same as HTTP content, so there is no disconnect. To enable older browsers to support caching, set the Cache-Control header.

res.header('Cache-Control', 'max-age=31536000');

The header above tells the browser to cache for one year.

The real issue comes with proxy caching. Proxy caching comes from an ISP or a service meant to speed up connections. This is mostly used in rural parts of the world that have slow internet connection speeds. Using HTTPS, this type of caching is impossible, because the proxy sees encrypted traffic. If you have a large global user base or an application that targets users in remote locations, you should consider this issue.

Another thing to think about: there are probably parts of your site that should NOT be cached. Do not let the browser cache everything. Plan which parts of your application should be cached and for how long. For example, CSS and JavaScript should be cached for a significant amount of time, while the user’s timeline view should update frequently.

Certificate types#

There are two types of SSL certificates.

Domain Validated Certificates are cheap but do not verify as much information as their counterparts. Usually starting around $50, they are the best option for small sites. From a user perspective, the downside is the distinction in the browser between different types of certificates. For example, a Domain Validated Certificate might only show a lock symbol in the address bar, while an Extended Validation Certificate will show the full green address bar. Additionally, check this lesson for information on Let’s Encrypt. They offer domain validated certificates free of charge.

Extended Validation Certificates are the gold standard of SSL certificates. They validate you as the domain’s owner and verify the identity and legitimacy of the domain owner. Because of this, these certificates are significantly more expensive. Extended Validation Certificates start at around $500 and are the certificate of choice for most large and reputable companies. Browsers display the full green address bar when an Extended Validation Certificate is in use, giving users peace of mind.
